
## Workspace

- This repo uses Bun workspaces, not pnpm. Install with `bun install`.
- Root scripts are limited to `bun run dev`, `bun run build`, and `bun run lint`; there is no root test script.
- Runtime targets are pinned in `package.json`: Node `>=22.18.0`, Bun `>=1.3.4`.

## Product Boundaries

- `apps/oc-pi-cli` is the product runtime and main implementation target.
- `apps/web-docs` is the current docs control plane for planning `apps/oc-pi-cli`; it is not the final user-facing generated project template.
- If product docs and code seem inconsistent, treat executable config and `apps/oc-pi-cli/src` as the implementation truth, then update docs to match.

## Web Docs

- `apps/web-docs` is a Next.js 16 + Fumadocs app.
- Fumadocs content source is hard-coded in `apps/web-docs/source.config.ts` as `content/docs`; do not move docs under `docs/` at repo root.
- The docs UI renders from `/docs/*`; the home page only redirects to `/docs`.
- Do not edit `apps/web-docs/.source/`; it is generated by `fumadocs-mdx`.
- Use these focused commands for docs work:
  - `bun --filter web-docs run dev` starts the docs app on port `4124`
  - `bun --filter web-docs run lint`
  - `bun --filter web-docs run types:check`
  - `bun --filter web-docs run build`

## Docs Source Of Truth

- The current docs tree contract is defined in `apps/web-docs/content/docs/planning/docs-structure-and-output-spec.md`.
- Before creating a new doc under `apps/web-docs/content/docs`, prefer updating the existing source-of-truth page for that topic.
- Directory roles are intentional:
  - `product/` for vision, roadmap, milestones, global status
  - `capabilities/` for capability map and boundaries
  - `architecture/` for runtime model and module structure
  - `planning/` for rules, protocols, and feature planning
  - `tasks/` for backlog, in-progress, next-up, and execution docs
  - `references/` for external notes that should not drive implementation directly
  - `protocols/` for cross-cutting docs rules
  - `archive/` for historical context only
- `apps/web-docs/content/docs/tasks/dev-tasks/` files should keep numeric ordering prefixes like `00-runtime-type-skeleton.md`.

## Docs Writing Rules

- For files under `apps/web-docs/content`, the first appearance of an English term must include a Chinese explanation.
- Relationship names must be written as `English + 中文语义`.
- Field names must include a Chinese definition.
- If a section title is technical jargon, the first sentence of the section should explain it in Chinese.
- Do not leave foundation/product docs with unexplained English-only terminology.
- The current canonical rule page is `apps/web-docs/content/docs/protocols/docs-writing-rule.md`; the same file under `archive/` is historical only.

## CLI And Artifact Flow

- `apps/oc-pi-cli/src/index.ts` is the CLI entrypoint.
- Focused CLI checks:
  - `bun --filter oc-pi-cli run dev`
  - `bun --filter oc-pi-cli run types:check`
- `goal new ... --write` requests artifact writes; in `development/test` it writes into `tests/sandbox`, and only packaged `production` runtime may write into `apps/web-docs`.
- `OC_PI_RUNTIME_STAGE` declares runtime purpose: `development` and `test` force sandbox writes, `production` enables real docs writes.
- Path guards in `apps/oc-pi-cli/src/runtime/paths.ts` intentionally refuse writes outside allowed roots for real docs output, sandbox output, and internal state files.
